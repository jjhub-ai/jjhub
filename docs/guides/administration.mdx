# Administration

JJHub includes a set of platform-wide administration tools for managing users, organizations, repositories, compute resources, feature flags, and audit logs. Admin operations are available through the CLI, API, and web interface.

## Who Has Admin Access

Admin access is granted through the `is_admin` flag on a user account. Admin users can access all `/api/admin/*` endpoints and `jjhub admin` CLI commands.

Admin API calls require a personal access token with the appropriate admin scope:

| Scope | Grants |
|-------|--------|
| `read:admin` | List users, orgs, repos, runners, audit logs, waitlist entries |
| `write:admin` | Create/delete users, approve waitlist entries, manage whitelist, modify admin flags |

Both scopes are enforced at the route level. A token without the required scope receives a `403 Forbidden` response even if the user is an admin.

## User Management

Admins can list, create, disable, and delete users across the platform.

### CLI

```bash
# List all users (paginated)
jjhub admin user list
jjhub admin user list --page 2 --limit 50

# Create a new user
jjhub admin user create --username alice --email alice@example.com

# Create a user with a specific password
jjhub admin user create --username bob --email bob@example.com --password "s3cret"

# Disable a user (sets active=false, blocks login and API access)
jjhub admin user disable alice

# Delete a user and all associated data
jjhub admin user delete alice
```

### API

```bash
# List users
curl https://api.jjhub.tech/api/admin/users \
  -H "Authorization: token jjhub_xxx"

# Create a user
curl -X POST https://api.jjhub.tech/api/admin/users \
  -H "Authorization: token jjhub_xxx" \
  -H "Content-Type: application/json" \
  -d '{
    "username": "alice",
    "email": "alice@example.com",
    "must_change_password": true
  }'

# Delete a user
curl -X DELETE https://api.jjhub.tech/api/admin/users/alice \
  -H "Authorization: token jjhub_xxx"

# Toggle admin flag on a user
curl -X PATCH https://api.jjhub.tech/api/admin/users/alice/admin \
  -H "Authorization: token jjhub_xxx" \
  -H "Content-Type: application/json" \
  -d '{"is_admin": true}'

# Create a token on behalf of a user
curl -X POST https://api.jjhub.tech/api/admin/users/alice/tokens \
  -H "Authorization: token jjhub_xxx" \
  -H "Content-Type: application/json" \
  -d '{"name": "ci-token", "scopes": ["read:repository", "write:repository"]}'
```

## Organization Management

Admins can list all organizations across the platform, regardless of membership.

### CLI

```bash
# List all organizations
jjhub api -X GET /admin/orgs
```

### API

```bash
# List all organizations
curl https://api.jjhub.tech/api/admin/orgs \
  -H "Authorization: token jjhub_xxx"
```

The response includes each organization's visibility, member count, and creation date. Admins can use the standard organization management endpoints (`/api/orgs/:org`) to modify settings on any organization without being a member.

## Repository Management

Admins can list all repositories across all users and organizations.

### CLI

```bash
# List all repositories platform-wide
jjhub api -X GET /admin/repos
```

### API

```bash
# List all repositories
curl https://api.jjhub.tech/api/admin/repos \
  -H "Authorization: token jjhub_xxx"
```

Admins have full access to any repository through the standard repository endpoints (`/api/repos/:owner/:repo`), including archiving, unarchiving, and deletion.

```bash
# Archive a repository
curl -X POST https://api.jjhub.tech/api/repos/alice/my-project/archive \
  -H "Authorization: token jjhub_xxx"

# Unarchive a repository
curl -X POST https://api.jjhub.tech/api/repos/alice/my-project/unarchive \
  -H "Authorization: token jjhub_xxx"

# Delete a repository
curl -X DELETE https://api.jjhub.tech/api/repos/alice/my-project \
  -H "Authorization: token jjhub_xxx"
```

## Compute and Workspace Controls

Admins can inspect compute runners and cross-repo workflow runs.

### Runners

```bash
# List all runners and their status
jjhub admin runner list
```

```bash
# API equivalent
curl https://api.jjhub.tech/api/admin/runners \
  -H "Authorization: token jjhub_xxx"
```

The runner list shows pool utilization, runner state, and capacity information.

### Workflow Runs (Cross-Repo)

Admins can list workflow runs across all repositories, filtered by status:

```bash
# List all workflow runs
jjhub admin workflow list

# Filter by status
jjhub admin workflow list --status running
jjhub admin workflow list --status failure --limit 10
```

Available status filters: `queued`, `running`, `success`, `failure`.

## Feature Flags

JJHub uses server-side feature flags to gate product areas. The flags are configured through environment variables with the `JJHUB_FEATURE_FLAGS_` prefix and exposed through a public API endpoint.

### Current Flags

| Flag | Description |
|------|-------------|
| `readout_dashboard` | Readout dashboard views |
| `landing_queue` | Programmable landing queue |
| `tool_skills` | Tool skill management |
| `tool_policies` | Tool policy configuration |
| `repo_snapshots` | Repository snapshot capture |
| `integrations` | Integration setup surfaces |
| `session_replay` | Session replay diagnostics |
| `secrets_manager` | Secrets management UI |
| `web_editor` | In-browser code editor |
| `client_error_reporting` | Client-side error reporting |
| `client_metrics` | Client-side performance metrics |

### Querying Flags

The feature flag endpoint is public (no authentication required) so that web and CLI clients can evaluate flags before establishing a session:

```bash
curl https://api.jjhub.tech/api/feature-flags
```

Response:

```json
{
  "flags": {
    "readout_dashboard": false,
    "landing_queue": false,
    "tool_skills": false,
    "tool_policies": false,
    "repo_snapshots": true,
    "integrations": true,
    "session_replay": false,
    "secrets_manager": true,
    "web_editor": false,
    "client_error_reporting": true,
    "client_metrics": true
  }
}
```

### Configuring Flags

Feature flags are set through server configuration (environment variables), not through the API. To enable or disable a flag, set the corresponding environment variable and restart the API server:

```bash
JJHUB_FEATURE_FLAGS_LANDING_QUEUE=true
JJHUB_FEATURE_FLAGS_SESSION_REPLAY=true
JJHUB_FEATURE_FLAGS_WEB_EDITOR=false
```

Feature-flagged routes in the web UI use `FlaggedRoute` guards that redirect users when a flag is disabled. The flags are enforced server-side as well -- API endpoints behind a flag return `404` when the flag is off.

## Audit Log Viewing

JJHub records audit events for security-sensitive operations. Admins can query the audit log to inspect who did what, when, and from where.

### Audit Log Fields

Each audit log entry contains:

| Field | Description |
|-------|-------------|
| `id` | Unique log entry ID |
| `event_type` | Category of the event |
| `actor_id` | User ID of the person who performed the action |
| `actor_name` | Username of the actor |
| `target_type` | Type of resource affected (user, repo, org, token, etc.) |
| `target_id` | ID of the affected resource |
| `target_name` | Name of the affected resource |
| `action` | Specific action taken (create, delete, update, login, etc.) |
| `metadata` | Additional context as JSON |
| `ip_address` | IP address of the request |

### Querying Audit Logs

```bash
# List audit logs since a date (required parameter)
curl "https://api.jjhub.tech/api/admin/audit-logs?since=2026-03-01" \
  -H "Authorization: token jjhub_xxx"

# With RFC3339 timestamp and pagination
curl "https://api.jjhub.tech/api/admin/audit-logs?since=2026-03-01T00:00:00Z&per_page=100" \
  -H "Authorization: token jjhub_xxx"
```

The `since` parameter is required and accepts either `YYYY-MM-DD` or full RFC3339 format. Results are paginated with a default of 50 entries per page.

## Alpha and Access Controls

JJHub uses a closed alpha model with a waitlist and whitelist for controlling access.

### Waitlist

Users can join the waitlist without authentication. Admins review and approve entries:

```bash
# Join the waitlist (no auth required)
jjhub waitlist join --email user@example.com --note "Interested in jj-native CI"

# List waitlist entries (admin)
jjhub waitlist list
jjhub waitlist list --status pending

# Approve a waitlist entry (admin)
jjhub waitlist approve --email user@example.com
```

### Whitelist

Admins can directly add identities to the whitelist to grant immediate access:

```bash
# Add an email to the whitelist
jjhub whitelist add --type email --value user@example.com

# Add a username
jjhub whitelist add --type username --value alice

# List whitelist entries
jjhub whitelist list

# Remove a whitelist entry
jjhub whitelist remove --type email --value user@example.com
```

### API

```bash
# List waitlist entries
curl "https://api.jjhub.tech/api/admin/alpha/waitlist?status=pending" \
  -H "Authorization: token jjhub_xxx"

# Approve a waitlist entry
curl -X POST https://api.jjhub.tech/api/admin/alpha/waitlist/approve \
  -H "Authorization: token jjhub_xxx" \
  -H "Content-Type: application/json" \
  -d '{"email": "user@example.com"}'

# Add to whitelist
curl -X POST https://api.jjhub.tech/api/admin/alpha/whitelist \
  -H "Authorization: token jjhub_xxx" \
  -H "Content-Type: application/json" \
  -d '{"identity_type": "email", "identity_value": "user@example.com"}'

# Remove from whitelist
curl -X DELETE https://api.jjhub.tech/api/admin/alpha/whitelist/email/user%40example.com \
  -H "Authorization: token jjhub_xxx"
```

## Service Health

The system health endpoint provides an operational overview of the platform:

```bash
# CLI
jjhub admin health

# API
curl https://api.jjhub.tech/api/admin/system/health \
  -H "Authorization: token jjhub_xxx"
```

The health check reports the status of core subsystems including the database, repo-host, workspace service, and queue.

## API Reference

All admin endpoints require the `RequireAdmin` middleware and the appropriate scope.

| Method | Endpoint | Scope | Description |
|--------|----------|-------|-------------|
| `GET` | `/api/admin/users` | `read:admin` | List all users |
| `POST` | `/api/admin/users` | `write:admin` | Create a user |
| `DELETE` | `/api/admin/users/:username` | `write:admin` | Delete a user |
| `PATCH` | `/api/admin/users/:username/admin` | `write:admin` | Set/unset admin flag |
| `POST` | `/api/admin/users/:username/tokens` | `write:admin` | Create token for user |
| `GET` | `/api/admin/orgs` | `read:admin` | List all organizations |
| `GET` | `/api/admin/repos` | `read:admin` | List all repositories |
| `GET` | `/api/admin/runners` | `read:admin` | List compute runners |
| `GET` | `/api/admin/audit-logs` | `read:admin` | Query audit logs |
| `GET` | `/api/admin/system/health` | `read:admin` | System health check |
| `GET` | `/api/admin/alpha/waitlist` | `read:admin` | List waitlist entries |
| `POST` | `/api/admin/alpha/waitlist/approve` | `write:admin` | Approve waitlist entry |
| `GET` | `/api/admin/alpha/whitelist` | `read:admin` | List whitelist entries |
| `POST` | `/api/admin/alpha/whitelist` | `write:admin` | Add whitelist entry |
| `DELETE` | `/api/admin/alpha/whitelist/:type/:value` | `write:admin` | Remove whitelist entry |
| `GET` | `/api/feature-flags` | (public) | Get feature flag values |
